## Clarifications

### Session 2026-05-23

- Q: 착용 기록 시 메모 입력 흐름을 어떤 형태로 채택할까요? → A: "착용 기록하기" 탭 시 가벼운 바텀시트가 떠서 한 줄 메모(선택) 입력 후 확정. 빈 입력으로 확정 시 현행 흐름과 동일.
- Q: 메모 입력을 어느 이벤트 종류까지 적용할까요? → A: 착용 이벤트(`kind=wear`)에만 적용. 세탁 완료 처리는 현행 다중 선택 일괄 흐름을 유지(메모 없음).
- Q: 메모 입력 길이/형식 제약은 어떻게 둘까요? → A: 단일행, 최대 80자, 빈 입력 허용(`note=null`).
- Q: 이미 기록된 이벤트의 메모를 사후 편집할 수 있어야 하나요? → A: 편집 + 삭제 모두 허용(타임라인 항목 long-press 또는 별도 액션).
- Q: 착용 이벤트 삭제 시 Item의 카운터·상태를 어떻게 처리할까요? → A: 사이드이펙트 전체 역적용·재평가 — `totalWears-1`, `wearSinceWash-1`, `lastWornAt` 재계산(남은 wear 이벤트 최신값 또는 null), 임계 미달 시 `dirty→clean` 자동 복귀.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 옷 등록 (Priority: P1)

큐레이터(사용자)는 새로 산 옷이나 이미 가지고 있는 옷을 자신의 디지털 옷장에 등록한다. 사진, 이름, 카테고리, 세탁 주기(몇 회 착용 후 세탁할지), 세탁 방법, 브랜드, 구매일을 입력하면 즉시 옷장 컬렉션에 추가된다.

**Why this priority**: 등록된 옷이 한 점도 없으면 옷장·세탁·홈 어느 화면도 의미를 갖지 못한다. 모든 후속 흐름의 전제 조건이며, 가장 먼저 동작해야 하는 MVP의 시작점이다.

**Independent Test**: 옷이 0점인 상태에서 등록 화면으로 진입해 사진/명칭/카테고리를 입력 후 저장하면 옷장 탭에 새 옷이 즉시 표시되는지 확인한다. 단독으로 가치(첫 옷 등록 성공)를 전달한다.

**Acceptance Scenarios**:

1. **Given** 옷장이 비어 있고 사용자가 옷장 탭 상단 + 버튼을 탭한 상태, **When** 명칭에 "오버사이즈 캐시미어 코트"를 입력하고 카테고리 "아우터"를 선택한 뒤 "등록하기"를 누른다, **Then** 옷장 탭의 그리드 최상단에 해당 옷이 표시되고 토스트로 "새 옷이 옷장에 등록됐어요"가 노출된다.
2. **Given** 등록 화면이 열려 있다, **When** 명칭 필드가 비어 있다, **Then** "등록하기"와 상단 "저장" 버튼은 비활성(흐림) 상태로 클릭이 무시된다.
3. **Given** 사용자가 세탁 주기 입력에서 "+" 버튼을 5회 탭한다, **When** 등록을 완료한다, **Then** 등록된 옷의 "세탁 주기"가 등록 시점의 카운트(예: 기본 5 + 5 = 10)로 저장되어 옷 상세에서 동일하게 표시된다.

---

### User Story 2 - 옷장 탐색 (Priority: P2)

사용자는 자신의 전체 옷장을 카테고리, 검색어, 정렬 기준으로 탐색한다. 각 옷 타일에서 사진·이름·세탁 후 착용 횟수·마지막 세탁일·청결 상태 라벨을 한눈에 확인한다.

**Why this priority**: 옷 등록 다음으로 자주 일어나는 활동이다. 사용자가 "지금 입을 옷"을 빠르게 찾는 핵심 유틸리티이며, 등록 데이터의 가치를 처음 체감하는 화면이다.

**Independent Test**: 최소 5점 이상의 옷이 등록된 상태에서 옷장 탭으로 진입, 카테고리 필터·검색·정렬을 각각 적용해 그리드가 즉시 갱신되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 다양한 카테고리의 옷 10점이 등록되어 있다, **When** 사용자가 카테고리 칩에서 "상의"를 선택한다, **Then** 그리드에는 카테고리가 "상의"인 옷만 표시된다.
2. **Given** 검색창에 "캐시미어"를 입력했다, **When** 일치하는 옷 이름·브랜드가 있다, **Then** 그리드는 일치 항목만 보여주고, 없으면 "검색 결과가 없습니다." 빈 상태 메시지가 노출된다.
3. **Given** 정렬 드롭다운이 "상태순(깨끗한 옷 먼저)"이다, **When** 그리드를 본다, **Then** `status = clean`인 옷이 `dirty`인 옷보다 위에 정렬된다.
4. **Given** 옷의 세탁 후 착용 횟수가 0인 옷이 있다, **When** 그리드를 본다, **Then** 해당 타일에는 좌상단에 녹색 톤 "깨끗함" 라벨이 표시되고, `dirty` 상태인 옷에는 "세탁 필요" 라벨이 표시된다.

### User Story 3 - 착용 기록 & 옷 상세 (Priority: P3)

사용자는 옷 타일을 탭해 상세 화면에서 큰 사진과 메타데이터(세탁 후 착용 횟수, 총 착용 횟수, 마지막 세탁일, 구매일)를 확인한다. "착용 기록하기"를 눌러 오늘의 착용을 기록하면 카운터가 올라가고, 세탁 주기를 초과하면 자동으로 `dirty` 상태로 전환된다.

**Why this priority**: 옷장 앱의 핵심 차별점이다. 단순한 옷 목록을 넘어 "언제 입고 언제 빨았는가"를 추적해 의류 관리 부담을 줄이는 가치를 만든다.

**Independent Test**: 등록된 옷의 상세에 진입해 "착용 기록하기"를 1회 누르면 세탁 후 착용 횟수가 +1 되고, 세탁 주기에 도달하면 옷장 그리드에서 동일 옷의 라벨이 "세탁 필요"로 바뀌는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 세탁 후 착용 횟수 2/5인 옷의 상세 화면이 열려 있다, **When** "착용 기록하기" 버튼을 누르고 떠오른 바텀시트에서 메모를 비워둔 채 "기록" 버튼을 누른다, **Then** 카운터는 3/5로 갱신되고 마지막 착용일이 "오늘"로 표시되며, 토스트로 "오늘의 착용이 기록되었어요"가 노출되고 `note=null`인 wear 이벤트가 타임라인에 추가된다. 사용자는 자동으로 직전 화면(옷장 또는 홈)으로 돌아간다.
2. **Given** "착용 기록하기" 바텀시트가 열려 있다, **When** 사용자가 메모 입력란에 "오피스 미팅, 따뜻함"(<=80자) 한 줄을 입력하고 "기록"을 누른다, **Then** 카운터는 +1 되고 해당 wear 이벤트는 입력된 메모와 함께 저장되며 타임라인 항목에 메모 본문이 노출된다.
3. **Given** 세탁 후 착용 횟수 4/5인 옷의 상세 화면이 열려 있다, **When** "착용 기록하기"를 누른 뒤 메모 시트에서 "기록"을 누른다, **Then** 카운터는 5/5가 되며 옷의 상태가 `dirty`로 전환되어 옷장 그리드에서 "세탁 필요" 라벨이 표시된다.
4. **Given** 옷 상세에서 "세탁 바구니" 보조 버튼을 누른다, **When** 해당 옷이 바구니에 없는 상태였다, **Then** 옷은 세탁 바구니에 추가되고 토스트로 "세탁 바구니에 담겼어요"가 노출되며 버튼 라벨이 "바구니에서 제외"로 바뀐다.
5. **Given** 옷 상세에서 착용 히스토리 타임라인이 표시된다, **When** 사용자가 화면을 본다, **Then** 최근의 착용·세탁 이벤트가 날짜 순(최신이 위)으로 도트와 함께 나열되며, wear 이벤트에 메모가 있으면 메모 본문이 함께 노출된다.
6. **Given** 타임라인에 wear 이벤트(메모 "오피스 미팅")가 있다, **When** 해당 항목을 길게 눌러 "메모 편집"을 선택하고 편집 시트에서 "외부 행사, 약간 더웠음"으로 수정해 저장한다, **Then** 해당 이벤트의 메모가 갱신되고 타임라인이 즉시 새 내용으로 반영된다. Item 카운터·상태는 변경되지 않는다.
7. **Given** 세탁 후 착용 횟수 3/5인 옷의 타임라인에 오늘자 wear 이벤트 1건이 있다, **When** 해당 항목을 길게 눌러 "기록 삭제"를 선택하고 확인한다, **Then** 이벤트가 제거되고 `totalWears`/`wearSinceWash`가 각각 -1 되며, `lastWornAt`이 남은 wear 이벤트 중 최신값(없으면 `null`)으로 재계산된다. 임계 도달로 `dirty`였던 옷이 임계 미달이 되면 `clean`으로 자동 복귀하고 옷장 그리드 라벨도 함께 갱신된다.

### User Story 4 - 세탁 바구니 일괄 처리 (Priority: P4)

사용자는 세탁 바구니에 담긴 옷들을 한번에 확인하고, 실제로 세탁한 항목들을 선택해 "세탁 완료 처리"한다. 완료 처리된 옷들은 착용 횟수가 0으로 초기화되고 `clean` 상태로 돌아간다.

**Why this priority**: 착용-세탁 사이클을 닫는 마지막 단계다. 이 흐름 없이는 옷 상태가 영구적으로 `dirty`로 남아 앱의 가치가 점진적으로 손상된다. P1~P3 완료 후 실제 일상 사용에서 반드시 필요해진다.

**Independent Test**: 옷 상세 또는 옷장에서 옷 3점을 세탁 바구니에 담은 뒤, 세탁 탭에서 그 중 2점만 선택해 "세탁 완료 처리"를 누르면 선택된 2점만 `clean` + 0/N으로 초기화되고 바구니에서 제거되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 세탁 바구니에 옷 4점이 담겨 있고 그 중 2점이 체크된 상태, **When** "선택 항목 세탁 완료 처리" 버튼을 누른다, **Then** 체크된 2점은 `clean` 상태 + 세탁 후 착용 횟수 0으로 초기화되어 바구니에서 제거되고, 토스트로 "2점의 세탁이 완료됐어요"가 노출된다.
2. **Given** 세탁 바구니에 옷 4점이 있다, **When** "모두 선택" 버튼을 누른다, **Then** 4점이 모두 체크 상태가 되며, 다시 누르면 모두 해제된다.
3. **Given** 선택된 항목이 0건이다, **When** 세탁 완료 처리 버튼을 본다, **Then** 버튼은 비활성 상태로 클릭이 무시된다.
4. **Given** 세탁 바구니가 비어 있다, **When** 세탁 탭에 진입한다, **Then** "세탁할 옷이 없어요." 빈 상태 메시지가 표시된다.

### User Story 5 - 홈 대시보드 & 환경 설정 (Priority: P5)

사용자는 홈 탭에서 옷장의 상태(전체 의류 수, 깨끗한/관리 필요 의류 수, 카테고리별 분포, 최근 입은 옷, 세탁 바구니 미리보기)를 한눈에 본다. 설정 탭에서 알림 토글(세탁 알림, 주간 리포트, 오래 입지 않은 옷)을 켜고 끈다.

**Why this priority**: 데이터를 요약·시각화하는 부가 가치층이다. 이 화면이 없어도 P1~P4만으로 핵심 가치는 전달되지만, 매일 앱을 여는 동기와 사용자 신뢰를 강화한다.

**Independent Test**: 5점 이상의 옷이 다양한 상태로 등록된 후 홈 탭으로 이동, 상단 통계 카운트가 실제 옷장 데이터와 일치하고 카테고리 카드 탭이 옷장 탭으로 해당 카테고리 필터를 적용해 이동하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 옷장에 10점(깨끗 7점, 관리 필요 3점)이 있다, **When** 홈 탭을 본다, **Then** "전체 아이템 10 / 깨끗한 의류 7 / 관리 필요 3" 통계가 표시된다.
2. **Given** 홈 화면에서 "상의" 카테고리 카드를 탭한다, **When** 화면이 전환된다, **Then** 옷장 탭이 활성화되고 카테고리 필터가 "상의"로 미리 적용된 상태로 진입한다.
3. **Given** 설정 탭에서 "세탁 알림" 토글이 On이다, **When** 다시 탭해 Off로 바꾼다, **Then** 토글 UI가 즉시 Off로 갱신되고 다음 앱 실행 시에도 Off 상태가 유지된다.

### Edge Cases

- 옷장이 비어 있을 때 옷장 탭과 홈 탭의 모든 통계는 0으로 표시되며 그리드는 빈 상태 메시지(또는 빈 그리드)로 노출된다.
- 카테고리가 "신발·가방·액세서리·원피스·운동복"인 옷은 홈의 2x2 카테고리 카드(아우터·상의·하의·기타)의 "기타" 버킷에 합산된다.
- 사용자가 옷을 등록할 때 사진을 첨부하지 않으면 기본 플레이스홀더 이미지가 사용된다.
- 같은 옷을 세탁 바구니에 담은 상태에서 "착용 기록하기"를 누르면 카운터는 증가하지만 바구니 상태는 유지된다(사용자가 명시적으로 토글하기 전까지).
- 세탁 주기를 1로 설정한 옷을 1회 착용하면 즉시 `dirty`로 전환된다.
- 착용 기록 메모 입력란이 비어 있는 채로 "기록"을 확정하면 `WearEvent.note`는 `null`로 저장되며 타임라인에는 메모 본문을 출력하지 않는다.
- 메모 입력란이 80자 한도에 도달하면 추가 입력이 차단되고 현재 글자 수 표시가 강조된다(80/80).
- 옷의 마지막 남은 wear 이벤트를 삭제하면 `Item.lastWornAt`은 `null`로 설정되어 옷장·홈 화면 모두 마지막 착용 표기를 "—"로 출력한다(FR-010b).
- wear 이벤트 삭제로 `wearSinceWash`가 0이 되어도, 세탁 바구니에 수동으로 담긴 상태는 자동 해제되지 않는다(`inLaundry`는 사용자 토글 전까지 유지).
- 검색 키워드는 대소문자를 구분하지 않으며 옷 이름과 브랜드 두 필드를 동시에 매칭한다.
- 앱을 종료했다 다시 열어도 옷 데이터, 세탁 바구니, 마지막 활성 탭, 설정 토글 상태가 모두 유지된다.

## Requirements *(mandatory)*

### Functional Requirements

**옷 등록 (US1)**

- **FR-001**: 시스템은 사용자가 의류 사진(촬영 또는 앨범에서 선택), 명칭(필수), 브랜드, 카테고리(필수), 세탁 주기(양의 정수), 세탁 방법(드라이클리닝·기계세탁·핸드워시 중 1), 구매일을 입력해 새 옷을 등록할 수 있어야 한다.
- **FR-002**: 시스템은 명칭이 비어 있는 동안 "등록하기"와 상단 "저장" 버튼을 비활성 상태로 표시해야 한다.
- **FR-003**: 시스템은 사진을 첨부하지 않은 등록도 허용해야 하며, 그 경우 기본 플레이스홀더 이미지가 자동으로 사용된다.
- **FR-004**: 등록 즉시 새 옷은 옷장 컬렉션의 최상단에 노출되어야 한다.

**옷장 탐색 (US2)**

- **FR-005**: 시스템은 카테고리 칩(전체·아우터·상의·하의·원피스·신발·가방·액세서리·운동복·기타)으로 옷장 그리드를 필터링할 수 있어야 한다.
- **FR-006**: 시스템은 옷 이름과 브랜드를 동시에 대소문자 무시로 매칭하는 검색을 제공해야 한다.
- **FR-007**: 시스템은 (1) 상태순(깨끗 우선), (2) 최근 착용순, (3) 착용 빈도순의 정렬 옵션을 제공해야 한다.
- **FR-008**: 각 옷 타일은 사진, 이름, "세탁 후 착용 횟수/세탁 주기", 마지막 세탁일을 표시해야 하며, 세탁 후 착용 횟수가 0이면 "깨끗함" 라벨을, `dirty` 상태이면 "세탁 필요" 라벨을 시각적으로 구분해 표시해야 한다.

**착용 기록 & 옷 상세 (US3)**

- **FR-009**: 시스템은 옷 상세에서 큰 의류 사진, 브랜드 라벨, 명칭, 카테고리·세탁 방법 칩, 세탁 후 착용 횟수, 총 착용 횟수, 마지막 세탁일, 구매일을 표시해야 한다.
- **FR-010**: "착용 기록하기" 액션은 가벼운 바텀시트를 띄워 사용자에게 선택적 한 줄 메모(단일행, 최대 80자, 빈 입력 허용)를 받은 뒤 확정 시 해당 옷의 세탁 후 착용 횟수와 총 착용 횟수를 각각 +1 하고 마지막 착용일을 오늘로 설정하며, 입력된 메모(빈 입력은 `null`)와 함께 `kind=wear` 이벤트를 저장해야 한다. 메모 입력은 wear 이벤트에만 적용하며, 세탁 완료 처리(US4 FR-016) 흐름에는 메모 입력 단계를 두지 않는다.
- **FR-010a**: 사용자는 타임라인의 wear 이벤트 항목에서 메모를 사후 편집할 수 있어야 한다. 편집 시 별도 시트가 떠 동일한 80자/단일행 제약을 따르며, 저장 시 해당 이벤트의 `note`만 갱신되고 Item의 카운터·상태·날짜는 변경되지 않는다.
- **FR-010b**: 사용자는 타임라인의 wear 이벤트 항목을 삭제할 수 있어야 한다. 삭제 시 시스템은 단일 트랜잭션으로 (1) 해당 이벤트 row를 제거하고, (2) 대상 Item의 `totalWears`와 `wearSinceWash`를 각각 -1 하며(0 미만으로 내려가지 않음), (3) `lastWornAt`을 남은 wear 이벤트 중 가장 최신의 `occurredAt`(없으면 `null`)으로 재계산하고, (4) `wearSinceWash < washCycle`이 되면 `dirty`였던 상태를 `clean`으로 자동 복귀시켜야 한다.
- **FR-011**: 세탁 후 착용 횟수가 세탁 주기에 도달하거나 초과한 옷은 자동으로 `dirty` 상태로 전환되어야 한다.
- **FR-012**: 옷 상세에서 사용자는 해당 옷을 세탁 바구니에 토글로 담거나 제외할 수 있어야 한다.
- **FR-013**: 옷 상세에는 해당 옷의 착용·세탁 이벤트가 날짜 역순(최신 우선) 타임라인으로 표시되어야 한다. wear 이벤트에 메모가 있는 경우 본문을 항목에 함께 표시하며, wear 이벤트 항목은 길게 누름(long-press) 또는 항목 액션을 통해 메모 편집·기록 삭제 메뉴를 노출해야 한다(FR-010a/010b).

**세탁 바구니 (US4)**

- **FR-014**: 시스템은 세탁 바구니에 담긴 옷의 사진, 이름, 카테고리, 세탁 방법, 진행률(세탁 후 착용 횟수/세탁 주기)을 리스트로 표시해야 한다.
- **FR-015**: 사용자는 바구니 내 항목을 개별 체크박스로 다중 선택하거나 "모두 선택"으로 일괄 선택할 수 있어야 한다.
- **FR-016**: "선택 항목 세탁 완료 처리"는 체크된 옷들의 세탁 후 착용 횟수를 0으로 초기화하고 상태를 `clean`으로 전환한 뒤 바구니에서 제거해야 한다.
- **FR-017**: 선택된 항목이 0건이면 세탁 완료 처리 버튼은 비활성 상태여야 한다.

**홈 대시보드 (US5)**

- **FR-018**: 홈 화면 상단 통계 영역은 전체 옷 수, 깨끗한 옷 수, 관리 필요 옷 수를 실시간으로 반영해야 한다.
- **FR-019**: 홈 화면 카테고리 카드(아우터·상의·하의·기타)는 각 카테고리의 옷 수를 표시하고, 탭 시 옷장 탭으로 해당 카테고리 필터가 미리 적용된 상태로 이동해야 한다.
- **FR-020**: 홈의 "최근 입은 옷" 섹션은 가장 최근에 착용된 옷 최대 2점을 노출해야 한다.
- **FR-021**: 홈의 "세탁 바구니" 섹션은 바구니에 담긴 옷 중 최대 3점의 미리보기와 진행률 바, 무작위 전문가 팁 1건을 표시해야 한다.

**전역 / 비기능 (Cross-cutting)**

- **FR-022**: 시스템은 옷 데이터, 세탁 바구니 상태, 마지막으로 선택한 탭, 알림 토글 설정을 로컬 저장소에 영속화하고 앱 재시작 시 복원해야 한다.
- **FR-023**: 시스템은 옷 등록·착용 기록·세탁 완료 처리 등 사용자 액션마다 비차단 토스트 알림(약 2초 자동 사라짐)을 제공해야 한다.
- **FR-024**: 모든 UI는 `.specify/memory/design.md`("The Digital Atelier")에 정의된 디자인 토큰(색상·타이포·간격·라운딩·노 라인 규칙)을 따라야 한다.
- **FR-025**: 옷 등록·옷장 탐색·착용 기록·세탁 처리·홈 조회 모든 핵심 동작은 네트워크 없이 동작해야 한다.

### Key Entities

- **Item (의류)**: 옷장의 1점 단위. 속성 — 고유 ID, 이름, 브랜드, 카테고리, 세탁 방법, 상태(`clean` | `dirty`), 세탁 주기(양의 정수), 세탁 후 착용 횟수, 총 착용 횟수, 마지막 착용일, 마지막 세탁일, 구매일, 사진 경로, 대표 색상(이미지 폴백용).
- **Laundry Basket (세탁 바구니)**: 세탁 대상으로 마킹된 Item ID들의 집합. 개별 Item은 바구니에 포함되거나 포함되지 않은 두 상태만 가진다.
- **WearEvent (착용·세탁 이벤트)**: 특정 Item에 대해 발생한 사건의 시간순 기록. `kind` 필드로 "착용(wear)" 또는 "세탁(wash)"을 구분하며 날짜와 선택적 메모(`note: String?`, 단일행, 최대 80자)를 가진다. 메모는 `kind=wear` 이벤트에만 사용자 입력이 적용되며(FR-010), `kind=wash` 이벤트는 본 릴리스에서 메모를 받지 않는다. wear 이벤트는 사후 메모 편집(FR-010a) 및 삭제(FR-010b, 사이드이펙트 역적용 포함)가 허용되며, 옷 상세의 타임라인을 구성한다.
- **User Preferences (사용자 환경설정)**: 알림 토글 상태(세탁 알림·주간 리포트·오래 입지 않은 옷), 액센트 색상, 마지막 활성 탭.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 옷장에 새 옷을 등록하는 동작을 4단계(옷장 → + → 명칭 입력 → 등록하기) 이내, 30초 이내에 완료할 수 있다.
- **SC-002**: 사용자는 옷장 탭 진입부터 특정 옷을 찾아 탭하기까지의 시간이 검색·필터 보조 없이도 5초 이내(20점 컬렉션 기준)여야 한다.
- **SC-003**: "착용 기록하기" 버튼 1회 누름으로 세탁 후 착용 횟수가 갱신되고 100ms 이내에 UI가 새 상태를 반영한다.
- **SC-004**: 옷장에 옷 100점이 등록된 상태에서도 옷장 탭의 필터·정렬·검색 결과가 200ms 이내에 표시된다.
- **SC-005**: 앱을 강제 종료한 뒤 재실행 시 옷 데이터, 세탁 바구니, 마지막 탭, 알림 설정이 100% 보존된다.
- **SC-006**: 옷 1점을 등록·착용·세탁 완료까지 전체 사이클을 사용자가 단독으로(설명 없이) 처음 사용에서 성공적으로 끝낼 수 있는 비율 90% 이상.
- **SC-007**: 네트워크가 끊긴 상태에서도 옷 등록·탐색·착용 기록·세탁 처리·홈 조회의 핵심 작업 5가지가 모두 정상 동작한다.

## Assumptions

- 디자인 패키지의 React 프로토타입(`index.html` + JSX 모듈)은 시각·인터랙션 명세의 단일 진실 원천(source of truth)이며, 실제 구현 기술 스택은 plan 단계에서 결정한다(헌법 II에 따라 디자인 토큰 일관성만 강제됨).
- 다국어는 v1 범위 밖이다. 한국어 UI만 지원한다(헌법 Product Constraints).
- 사용자는 단일 사용자 1명이며 다중 사용자·계정 시스템은 v1에 없다(헌법 III, 로컬 우선).
- 사진은 단순화를 위해 카메라 촬영 또는 디바이스 앨범에서의 선택만 지원하고, 클라우드 사진 라이브러리 직접 연동은 범위 밖이다.
- 세탁 알림의 실제 푸시 발송 메커니즘은 v1 범위에서 토글만 노출하며, 실제 알림 스케줄링 동작은 별도 후속 spec으로 정의한다.
- 데이터 백업(iCloud·Google Drive)은 설정 화면의 진입점만 표시하고, 실제 백업·복원 동작은 별도 후속 spec으로 정의한다.
- 디자인 패키지에 포함된 샘플 데이터(예: ZARA 코트, A.P.C. 데님 등)는 개발용 시드 데이터로만 사용하며 사용자에게 사전 등록된 상태로 노출되지 않는다.
